# 工具调用失败问题根本原因分析

## 问题概述

从API历史记录中观察到大量工具调用失败,所有失败的错误信息为:
```
Invalid tool call for 'update_todo_list': missing nativeArgs. This usually means the model streamed invalid or incomplete arguments and the call could not be finalized.
```

类似地,其他工具如 `list_files`、`codebase_search` 也出现相同的错误。

## 问题现象

### 失败的工具调用示例

1. **update_todo_list** - 空参数对象
```json
{
  "type": "tool_use",
  "id": "call_0a919e3666004ee8aacbc345",
  "name": "update_todo_list",
  "input": {}
}
```

2. **list_files** - 空参数对象
```json
{
  "type": "tool_use",
  "id": "call_e616456e58254bc89067c8fb",
  "name": "list_files",
  "input": {}
}
```

3. **codebase_search** - 空参数对象
```json
{
  "type": "tool_use",
  "id": "call_4f39a8ac38ea47a9bae36671",
  "name": "codebase_search",
  "input": {
    "0": "command auto approve",
    "1": "auto reject command",
    "2": "command deny list"
  }
}
```

### 共同特征

所有失败的工具调用都具有以下特征:
- `input` 字段要么是空对象 `{}`
- 要么包含数字键的参数(如 `"0"`, `"1"`, `"2"`)
- 缺少 `nativeArgs` 字段
- 错误信息一致: "missing nativeArgs"

## 根本原因分析

### 1. 工具调用处理流程

工具调用的处理流程如下:

```
API响应 → 流式处理 → NativeToolCallParser → presentAssistantMessage → 工具执行
```

### 2. 核心问题:参数格式不匹配

#### 问题1:流式参数累积失败

在 `src/core/task/streaming/handlers/ToolCallHandler.ts` 中,`handleDirectToolCallDelta` 函数负责处理流式工具调用参数更新:

```typescript
// Line 290-327
async function handleDirectToolCallDelta(
  context: ChunkHandlerContext,
  chunk: Extract<StreamChunk, { type: "tool_call_delta" }>
): Promise<void> {
  // ...
  let paramsUpdate: any = {}
  try {
    // 尝试解析delta为JSON
    const parsed = JSON.parse(chunk.delta)
    paramsUpdate = parsed
  } catch {
    // 如果不是有效JSON,作为部分JSON字符串累积
    const existingToolUse = context.stateManager.getAssistantMessageContent()[toolUseIndex]
    if (existingToolUse && existingToolUse.type === "tool_use") {
      const currentParams = existingToolUse.params || {}
      const currentParamsStr = JSON.stringify(currentParams)
      const newParamsStr = currentParamsStr + chunk.delta

      try {
        paramsUpdate = JSON.parse(newParamsStr)
      } catch {
        // 仍然是不完整的JSON,保持原样
        existingToolUse.params = currentParams
        context.config.onPresentAssistant()
        return  // 关键:这里直接返回,不更新nativeArgs
      }
    }
  }
  // 更新工具使用
  existingToolUse.params = { ...existingToolUse.params, ...paramsUpdate }
  // ...
}
```

**问题**:当JSON解析失败时,函数直接返回,但此时 `existingToolUse.params` 可能是无效的或空对象。

#### 问题2:nativeArgs构建缺失

在 `src/core/assistant-message/NativeToolCallParser.ts` 中,`parseToolCall` 方法负责构建 `nativeArgs`:

```typescript
// Line 717-1079
public static parseToolCall<TName extends ToolName>(toolCall: {
  id: string
  name: TName
  arguments: string
}): ToolUse<TName> | McpToolUse | null {
  // ...
  try {
    // 解析参数JSON字符串
    const args = toolCall.arguments === "" ? {} : JSON.parse(toolCall.arguments)

    // 构建字符串化参数用于显示/日志
    const params: Partial<Record<ToolParamName, string>> = {}

    for (const [key, value] of Object.entries(args)) {
      // 验证参数名称
      if (!toolParamNames.includes(key as ToolParamName) && !customToolRegistry.has(resolvedName)) {
        console.warn(`Unknown parameter '${key}' for tool '${resolvedName}'`)
        continue
      }
      // 转换为字符串
      const stringValue = typeof value === "string" ? value : JSON.stringify(value)
      params[key as ToolParamName] = stringValue
    }

    // 构建类型化的nativeArgs
    let nativeArgs: NativeArgsFor<TName> | undefined = undefined

    switch (resolvedName) {
      case "update_todo_list":
        // 注意:这里没有处理update_todo_list的参数构建逻辑
        // 导致nativeArgs始终为undefined
        break
      case "list_files":
        // 注意:这里也没有处理list_files的参数构建逻辑
        break
      // ...
    }

    // Native-only:核心工具必须具有类型化的nativeArgs
    if (!nativeArgs && !customToolRegistry.has(resolvedName)) {
      throw new Error(
        `[NativeToolCallParser] Invalid arguments for tool '${resolvedName}'. ` +
        `Native tool calls require a valid JSON payload matching the tool schema. ` +
        `Received: ${JSON.stringify(args)}`
      )
    }
    // ...
  } catch (error) {
    console.error(`Failed to parse tool call arguments: ${error}`)
    return null
  }
}
```

**问题**:对于某些工具(如 `update_todo_list`, `list_files`, `codebase_search`),`parseToolCall` 的 switch 语句中缺少对应的 case,导致 `nativeArgs` 无法被构建。

#### 问题3:参数验证失败

在 `presentAssistantMessage.ts` 中,工具执行前会检查 `nativeArgs`:

### 3. 具体工具的问题分析

#### update_todo_list 工具

从工具定义看,`update_todo_list` 需要一个 `todos` 参数:


3. 参数验证失败,无法构建 `nativeArgs`

### 4. 流式处理与参数累积的问题

在流式处理中,工具调用的参数是逐步累积的:

```
tool_call_start → tool_call_delta → tool_call_delta → ... → tool_call_end
```

**问题场景**:

1. **场景1**:模型流式输出参数时,JSON格式不完整
   - 例如:输出 `{ "todos": [ { "content": "探索项目...` 然后流结束
   - `handleDirectToolCallDelta` 无法解析为有效JSON,直接返回
   - `params` 保持为空对象 `{}`
   - `parseToolCall` 无法构建 `nativeArgs`

2. **场景2**:模型输出错误的参数格式
   - 例如:输出 `{ "0": "command auto approve", "1": "auto reject" }` 而不是 `{ "query": "command auto approve" }`
   - 参数名称不匹配,被过滤掉
   - `params` 变为空对象
   - `parseToolCall` 无法构建 `nativeArgs`

3. **场景3**:工具定义与实际调用不匹配
   - 某些工具的 `parseToolCall` 没有实现对应的 case
   - 即使参数正确,也无法构建 `nativeArgs`

## 解决方案

### 1. 补充缺失的工具参数构建逻辑

在 `src/core/assistant-message/NativeToolCallParser.ts` 中,为所有工具添加对应的 case:

```typescript
switch (resolvedName) {
  // 现有的 case...

  case "update_todo_list":
    if (args.todos !== undefined) {
      nativeArgs = {
        todos: args.todos
      } as NativeArgsFor<TName>
    }
    break

  case "list_files":
    if (args.path !== undefined) {
      nativeArgs = {
        path: args.path,
        recursive: args.recursive
      } as NativeArgsFor<TName>
    }
    break

  case "codebase_search":
    if (args.query !== undefined) {
      nativeArgs = {
        query: args.query
      } as NativeArgsFor<TName>
    }
    break

  // 其他工具...
}
```

### 2. 改进流式参数累积逻辑

在 `src/core/task/streaming/handlers/ToolCallHandler.ts` 中,改进 `handleDirectToolCallDelta`:

```typescript
async function handleDirectToolCallDelta(
  context: ChunkHandlerContext,
  chunk: Extract<StreamChunk, { type: "tool_call_delta" }>
): Promise<void> {
  const toolUseIndex = context.stateManager.getToolCallIndex(chunk.id)

  if (toolUseIndex === undefined) {
    console.warn(
      `[Task#${context.config.taskId}] Received tool_call_delta without tool_call_start for ID: ${chunk.id}`
    )
    return
  }

  let paramsUpdate: any = {}
  try {
    // 尝试解析delta为JSON
    const parsed = JSON.parse(chunk.delta)
    paramsUpdate = parsed
  } catch {
    // 如果不是有效JSON,作为部分JSON字符串累积
    const existingToolUse = context.stateManager.getAssistantMessageContent()[toolUseIndex]
    if (existingToolUse && existingToolUse.type === "tool_use") {
      const currentParams = existingToolUse.params || {}
      const currentParamsStr = JSON.stringify(currentParams)
      const newParamsStr = currentParamsStr + chunk.delta

      try {
        paramsUpdate = JSON.parse(newParamsStr)
      } catch {
        // 仍然是不完整的JSON,保持原样
        // 但需要记录累积的参数字符串
        if (!existingToolUse.argumentsAccumulator) {
          existingToolUse.argumentsAccumulator = ""
        }
        existingToolUse.argumentsAccumulator += chunk.delta
        context.config.onPresentAssistant()
        return
      }
    }
  }

  // 更新工具使用
  const existingToolUse = context.stateManager.getAssistantMessageContent()[toolUseIndex]
  if (existingToolUse && existingToolUse.type === "tool_use") {
    existingToolUse.params = { ...existingToolUse.params, ...paramsUpdate }
    context.stateManager.updateAssistantContentBlock(toolUseIndex, existingToolUse)

    // Update tool call token count
    const beforeTokens = context.tokenManager.getTotalEstimatedTokens()
    context.tokenManager.addToolCallTokens(
      chunk.id,
      existingToolUse.name,
      JSON.stringify(existingToolUse.params || {})
    )
    const afterTokens = context.tokenManager.getTotalEstimatedTokens()
    const delta = afterTokens - beforeTokens
    const breakdown = context.tokenManager.getTokenBreakdown()

    if (delta !== 0) {
      console.log(
        `[ToolCallHandler#handleDirectToolCallDelta] id=${chunk.id} name=${existingToolUse.name} | delta=${delta} | total=${afterTokens} | ` +
        `breakdown{text:${breakdown.text},reasoning:${breakdown.reasoning},toolCalls:${breakdown.toolCalls}}`
      )
    }
  }

  // Present assistant message
  context.config.onPresentAssistant()

  // Publish tool call progress event
  await context.eventBus?.publish('tool:call:progress', {
    toolCallId: chunk.id,
    progress: {
      percentage: undefined,
      message: undefined,
      details: {
        delta: chunk.delta,
        params: paramsUpdate,
      },
    },
  })
}
```

### 3. 改进工具调用完成处理

在 `handleDirectToolCallEnd` 中,使用累积的参数字符串:

```typescript
async function handleDirectToolCallEnd(
  context: ChunkHandlerContext,
  chunk: Extract<StreamChunk, { type: "tool_call_end" }>
): Promise<void> {
  const toolUseIndex = context.stateManager.getToolCallIndex(chunk.id)

  if (toolUseIndex === undefined) {
    const errorMessage = `Received tool_call_end without tool_call_start for ID: ${chunk.id}`
    console.warn(`[ToolCallHandler] ${errorMessage}`)

    // Publish tool call error event
    await context.eventBus?.publish('tool:call:error', {
      toolCallId: chunk.id,
      error: new Error(errorMessage),
      isRetryable: false,
    })
    return
  }

  // Finalize the tool use
  const existingToolUse = context.stateManager.getAssistantMessageContent()[toolUseIndex]
  if (existingToolUse && existingToolUse.type === "tool_use") {
    // 如果有累积的参数字符串,尝试解析
    if (existingToolUse.argumentsAccumulator) {
      try {
        const parsedArgs = JSON.parse(existingToolUse.argumentsAccumulator)
        existingToolUse.params = parsedArgs
      } catch (error) {
        console.error(`Failed to parse accumulated arguments for tool ${existingToolUse.name}:`, error)
        // 保持现有的params
      }
    }

    existingToolUse.partial = false
    ;(existingToolUse as any).id = chunk.id
    context.stateManager.updateAssistantContentBlock(toolUseIndex, existingToolUse)
  }

  context.stateManager.removeToolCallIndex(chunk.id)

  // Present assistant message
  context.config.onPresentAssistant()

  // Publish tool call complete event
  await context.eventBus?.publish('tool:call:complete', {
    toolCallId: chunk.id,
    result: {
      toolCallId: chunk.id,
      result: existingToolUse,
      success: true,
    },
  })
}
```

### 5. 添加工具参数验证

添加一个工具参数验证函数,在工具调用前验证参数:

```typescript
function validateToolParams(toolName: ToolName, args: any): { valid: boolean; error?: string } {
  const toolSchema = getToolSchema(toolName)
  if (!toolSchema) {
    return { valid: false, error: `Unknown tool: ${toolName}` }
  }

  // 检查必需参数
  const missingParams = toolSchema.required?.filter((param: string) => !(param in args))
  if (missingParams && missingParams.length > 0) {
    return {
      valid: false,
      error: `Missing required parameters for ${toolName}: ${missingParams.join(', ')}`
    }
  }

  // 检查参数类型
  for (const [paramName, paramSchema] of Object.entries(toolSchema.properties || {})) {
    if (paramName in args) {
      const value = args[paramName]
      const expectedType = paramSchema.type

      if (expectedType === "array" && !Array.isArray(value)) {
        return {
          valid: false,
          error: `Parameter '${paramName}' for ${toolName} should be an array, got ${typeof value}`
        }
      }
      // 其他类型检查...
    }
  }

  return { valid: true }
}
```

## 总结

### 问题根源

1. **代码缺失**: `NativeToolCallParser.parseToolCall` 中缺少某些工具的参数构建逻辑
2. **流式处理缺陷**: 参数累积机制不完善,无法正确处理不完整的JSON
3. **参数格式错误**: 模型输出的参数格式与工具定义不匹配
4. **验证过于严格**: 缺少 `nativeArgs` 就直接拒绝工具调用,没有尝试恢复

### 影响范围

- 影响所有使用 `parseToolCall` 而没有实现对应 case 的工具
- 主要影响: `update_todo_list`, `list_files`, `codebase_search` 等
- 导致用户体验差,工具调用频繁失败

### 修复优先级

1. **高优先级**: 补充所有缺失工具的参数构建逻辑
2. **中优先级**: 改进流式参数累积机制
3. **低优先级**: 改进错误处理和日志

### 测试建议

1. 为每个工具添加单元测试,覆盖各种参数场景
2. 添加流式处理的集成测试
3. 添加参数验证的测试用例
4. 测试模型输出错误参数格式的处理
